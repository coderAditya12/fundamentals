# What is Bytecode?

Bytecode is an **intermediate representation** of a Python program generated by the Python compiler.

It is **not machine code**.

Instead, it is a set of instructions specifically designed for the **Python Virtual Machine (PVM)**.

Execution Flow:

```text
Python Source Code (.py)
          │
          ▼
      Python Compiler
          │
          ▼
       Bytecode
          │
          ▼
Python Virtual Machine
          │
          ▼
 Machine Code Execution
```

---

# Why Does Python Use Bytecode?

Instead of compiling directly into machine code, Python compiles into bytecode because it provides:

* Platform independence
* Faster program startup
* Easier debugging
* Separation of compilation and execution
* Support for multiple Python implementations

---

# How Bytecode is Generated

Suppose we have the following program.

```python
def add(a, b):
    return a + b

print(add(10, 20))
```

When Python executes the program, the following steps occur.

```text
Source Code

↓

Tokenizer

↓

Parser

↓

AST

↓

Compiler

↓

Bytecode
```

The generated bytecode is then executed by the Python Virtual Machine.

---

# Viewing Bytecode

Python provides the **dis** module for inspecting bytecode.

Example

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
```

Possible Output

```text
LOAD_FAST 0 (a)

LOAD_FAST 1 (b)

BINARY_OP +

RETURN_VALUE
```

# Understanding Bytecode Instructions

Consider

```python
x = 10
```

Internally Python generates something similar to

```text
LOAD_CONST 10

STORE_NAME x
```

Meaning

```text
LOAD_CONST

↓

Load constant value 10

↓

STORE_NAME

↓

Store value inside variable x
```

---

Another example

```python
print("Hello")
```

Bytecode may look similar to

```text
LOAD_NAME print

LOAD_CONST "Hello"

CALL

POP_TOP

RETURN_VALUE
```

The PVM reads each instruction one by one.

# Where is Bytecode Stored?

CPython stores compiled bytecode inside

```text
__pycache__/
```

Example

```text
project/

main.py

__pycache__/

main.cpython-312.pyc
```

The filename tells us

```text
main

↓

Source file

cpython

↓

Implementation

312

↓

Python Version 3.12
```

---

# Why Python Creates .pyc Files

Imagine a project containing 500 Python files.

Without caching

```text
Run Program

↓

Compile 500 Files

↓

Execute
```

Every execution would require recompiling every module.

Instead Python caches bytecode.

Next execution becomes

```text
Run Program

↓

Use Existing Bytecode

↓

Execute
```

Result

* Faster startup
* Less CPU work

---

# When Does Python Recompile?

Python recompiles a module when

* The source file changes
* The `.pyc` file is missing
* The cached bytecode is incompatible with the current Python version

Otherwise Python simply reuses the existing bytecode.

---

# Is Bytecode Platform Independent?

Mostly yes.

Bytecode is **independent of the operating system and CPU architecture**, but it is **dependent on the Python implementation and version**.

For example

✔ CPython 3.12 Bytecode

Can generally run on

* Windows
* Linux
* macOS

provided they all use **CPython 3.12**.

However,

A `.pyc` generated by CPython may **not** work on PyPy or Jython.

Likewise, a `.pyc` generated by CPython 3.12 is not guaranteed to work with CPython 3.13.

# Bytecode vs Machine Code

| Feature              | Bytecode                    | Machine Code                  |
| -------------------- | --------------------------- | ----------------------------- |
| Executed By          | Python Virtual Machine      | CPU                           |
| Human Readable       | No                          | No                            |
| Platform Independent | Mostly                      | No                            |
| Generated By         | Python Compiler             | C/C++ Compiler                |
| Purpose              | Intermediate Representation | Final Executable Instructions |

---

# Why Doesn't Python Produce Machine Code?

If Python produced machine code directly,

it would need different executables for

* Windows
* Linux
* macOS
* ARM
* x86
* x64

Instead,

```text
Python Source

↓

Bytecode

↓

Platform-specific Interpreter
```

This greatly simplifies portability.

---

# Advantages of Bytecode

* Faster execution after first compilation
* Platform independent (within compatible implementations and versions)
* Easier debugging
* Simpler interpreter design
* Reusable compiled modules

---

# Disadvantages

* Slower than native machine code
* Requires a Python Virtual Machine
* Implementation-specific
* Cannot run directly on the CPU

---

# Common Misconceptions

### ❌ Bytecode is machine code.

Incorrect.

Bytecode is executed by the Python Virtual Machine.

Machine code is executed directly by the CPU.

---

### ❌ `.pyc` files make Python a compiled language.

Not exactly.

Python is generally described as a **compiled-and-interpreted language**.

It compiles source code into bytecode, then interprets that bytecode.

---

### ❌ Bytecode is universal.

Incorrect.

Bytecode compatibility depends on the Python implementation and version.
